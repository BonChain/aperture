# Story 3.3: Verify Proof Without Secret Key

Status: ready-for-dev

## Story

As a Verifier,
I want to verify a holder's proof against their public key and on-chain ciphertext without their secret key,
So that I get an unambiguous pass/fail.

## Acceptance Criteria

**AC-1 (off-chain + on-chain, fixture-driven):** Given a proof + holder pubkey + on-chain ciphertext(s), when the Verifier verifies, then it passes WITHOUT the holder's secret key — off-chain AND on-chain (`aperture::verifier::verify_aggregate`) — tested against the committed golden aggregate proof fixtures (`proofAggregateValid.hex` / `proofAggregateTampered.hex` from Story 1.1b), not Story 3.2's runtime output. *(FR-17, AR-6)*

**AC-2 (correct + tampered verdict):** Given a correct proof and matching `X`, then result = `badge-verified` "Verified"; given a tampered proof or mismatched input, then result = `badge-failed` "Doesn't verify" — unambiguous in both directions. *(FR-17, NFR-3, UX-DR7)*

**AC-3 (accessibility):** Given the verdict renders, then it carries text + icon, never color alone. `badge-verified` = emerald + checkmark icon + "Verified" text; `badge-failed` = rose + X icon + "Doesn't verify" text. *(UX-DR21)*

**AC-4 (done definition):** Story 3.3 is not done until the on-chain `verify_aggregate` call passes on devnet for the valid fixture AND aborts with code 100 for the tampered fixture. Off-chain alone is insufficient. *(AR-15 rule 5)*

**AC-5 (devnet degradation):** Given the devnet is unreachable, then the verify UI shows the off-chain result only, with a notice: "On-chain verification unavailable — showing off-chain result only." No crash, no white screen. *(UX-DR20)*

## Dev Agent Technical Context

### Two verify paths — both must pass

**Path A — Off-chain verify (synchronous, <10ms)**

The `verifyElGamal` function is implemented in `packages/spike/src/spike1.elgamal.test.ts` (lines 54–93) and its aggregate counterpart in `packages/spike/src/spike1.aggregate.test.ts`. The function is not yet exported from `packages/core` — the dev agent must either:
- Extract and export it from `packages/core/src/crypto/` (preferred, keeps crypto behind the right boundary), OR
- Inline a copy inside the verify UI's adapter layer

The function signature (from `spike1.elgamal.test.ts`):
```ts
function verifyElGamal(
  dst: Uint8Array,
  pkBytes: Uint8Array,
  ciphertextBytes: Uint8Array,
  decryptionHandleBytes: Uint8Array,
  proof: Uint8Array,
): boolean
```

The proof layout is `a ‖ b ‖ z1 ‖ z2` (4 × 32 bytes = 128 bytes total). For aggregate verification, the ciphertext and decryption handle are the component-wise Ristretto sum of all individual entry pairs — the verify function signature itself is identical.

**Path B — On-chain verify (async, ~4.5s)**

Calls `aperture::verifier::verify_aggregate` on devnet via PTB. The Move function aborts with `E_VERIFY_FAILED` (code 100) on false — it does NOT return bool. Copy the exact pattern from `packages/spike/test/onchain/onchain.devnet.test.ts`.

Key details from the on-chain test:
- `buildVerifyTx(packageId, statement, proof, amount)` constructs the PTB (see lines 60–84 of `onchain.devnet.test.ts`)
- The proof is passed split: `proof.slice(0, 32)` (a), `proof.slice(32, 64)` (b), `proof.slice(64, 96)` (z1), `proof.slice(96, 128)` (z2)
- The client is `new SuiJsonRpcClient({ url: getJsonRpcFullnodeUrl("devnet"), network: "devnet" })`
- The keypair is read from `~/.sui/sui_config/sui.keystore` using `readActiveKeypair()` (lines 103–136)
- Success → `finalized.effects?.status?.status === "success"`
- Abort → `signAndExecuteTransaction` rejects with `/abort code: 100/`

For the UI context (browser), the keypair is the holder's session-derived key — wire this via the existing key lifecycle (`apps/web/src/lib/keys.ts` if it exists, or the session key from RoleSwitcher context). The on-chain call belongs in a server-side route (`POST /proofs/verify`, per AR-16) or a dedicated API call from the web — do NOT put `~/.sui/sui_config/sui.keystore` reads into the browser. See architectural note below.

### Architectural note on the on-chain verify call

Architecture (AR-6, AR-16) specifies `POST /proofs/verify` as a backend route. The web UI submits `{ proof, pubkey, ciphertext, decryptionHandle, amount }` and the backend:
1. Runs off-chain verify first (cheap, synchronous)
2. If off-chain passes, runs on-chain verify (submits the PTB from the funded devnet address in the server keystore)
3. Returns `{ offChain: boolean, onChain: "success" | "abort:100" | "unreachable" }`

The UI maps these to badge-verified / badge-failed / degraded notice per AC-2 and AC-5.

If for time/scope reasons you wire the on-chain call client-side directly (using the `@mysten/sui` SDK in the browser with the holder's derived keypair), that is acceptable for the PoC — but document the deviation in a code comment. The off-chain half must always run in `packages/core/crypto` (AR-15 rule 1, lint zone).

**Story 3.3 primary test target = `proofAggregateValid.hex` / `proofAggregateTampered.hex`** (the aggregate path is the live path per SPIKE-1 GO). The single-amount fixtures (`proofValid.hex` / `proofTampered.hex`) serve as a sanity baseline.

### The aggregate statement

The aggregate statement for the `proofAggregateValid.hex` fixture (must match exactly):
```ts
// From onchain.devnet.test.ts + spike1.aggregate.test.ts
const AGG_SK = 12345n;
const ENTRY_A_AMOUNT = 40000n, ENTRY_A_BLINDING = 11111n;
const ENTRY_B_AMOUNT = 30000n, ENTRY_B_BLINDING = 22222n;
const AGG_AMOUNT = 70000n; // 70000 > 2^16

// Statement construction (component-wise Ristretto sum):
const pk = G.multiply(AGG_SK);
const c1A = G.multiply(ENTRY_A_BLINDING).add(H.multiply(ENTRY_A_AMOUNT));
const dhA = pk.multiply(ENTRY_A_BLINDING);
const c1B = G.multiply(ENTRY_B_BLINDING).add(H.multiply(ENTRY_B_AMOUNT));
const dhB = pk.multiply(ENTRY_B_BLINDING);
const statement = {
  dst: new Uint8Array(0),
  pk: pk.toBytes(),
  ciphertext: c1A.add(c1B).toBytes(),      // Ciphertext.add
  decryptionHandle: dhA.add(dhB).toBytes(), // Ciphertext.add
};
```

`H` is the secondary Ristretto generator: `34ce1477c14558178089500a39c864e0f607b3c1f41ab398400e4a9de6d2c446`
